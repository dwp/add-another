‘Add another’ patterns to test with screen reader users

Prototypes

  1. Single page pattern (based on MOJ example)
  2. Looping pattern

Notes from user research

We tested a couple of versions of the ‘add another’ pattern on 9 October 2019 with someone who is visually impaired and uses the screen reader, JAWS.

Version 1 - Single page (based on MOJ pattern)

The user heard ‘You can add more than one condition if you need to’ and took this to mean he could add multiple conditions into the ‘Name of condition’ field. He suggested changing the field label to something like ‘First condition’ to make it clearer. He realised this error when he got to the ‘Add another condition’ button.

Automatically focusing on the field after clicking the ‘add another condition’ button seemed to work well. It was clear to him where he was on the page.

When asked to remove a condition, he was able to navigate back through his answers to the right condition and complete the task, as the ‘remove’ button was in a logical place.

Automatically focusing on the H1 after removing a condition seemed to work well as it was clear to him where he was on the page.

He entered the full date in the day field to begin with. He realised this mistake though and corrected it.

The fact that there’s no confirmation message after something had been removed didn’t cause an issue.

Version 2 - Looping pattern

The ‘If you have more than one condition, you will be able to tell us about it on the next screen.’ made sense - he knew that we only needed to add one condition.

The visually hidden text in links and buttons was useful and made it quicker for him to understand what these would do without needing to hear the content that came before the link (‘delete broken leg’ etc).

There seemed to be an issue when changing a condition. For some reason, he was taken to the bottom of the screen after clicking ‘change’. We tried this a few times and this seemed to be happening consistently so is something to investigate (not sure why it’s happening now).

The fact that no confirmation message after something had been removed didn’t cause an issue.

Conclusion

Both prototypes could be made to work for this user with a bit of tweaking.

The first one would need a fall-back if JavaScript is unavailable / switched off, which would need to be tested separately.

The second version is a bit more verbose and takes longer to input the conditions, but is arguably clearer due to the use of the hidden text in the links/buttons. Work needs to be done to address the issue of the user being taken to the bottom of the screen though.
